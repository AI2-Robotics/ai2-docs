## ArmFactory {#ArmFactory}

The `ArmFactory` class provides the following methods for creating robotic arm instances.

1. Create a robotic arm instance by specifying the type and name.

   - Method:
   
     ```python
     create_arm(vendor_type: str, name: str, params: dict[str, str]) -> ArmInterface
     ```

   - Parameters:

     <table>
     <thead>
     <tr>
     <th>Parameter</th>
     <th>Description</th>
     </tr>
     </thead>
     <tbody>
     <tr>
     <td>
     `vendor_type`
     </td>
     <td>     
     The manufacturer or type of the robotic arm.    
     </td>
     </tr>
     <tr>
     <td>
     `name`
     </td>
     <td>
     The instance identifier for the arm (e.g., `"left_arm"`, `"right_arm"`).
     </td>
     </tr>
     <tr>
     <td>
     
     `params`
     
     </td>
     <td>
     
     Additional robotic arm parameters, such as IP address, port number, etc.
     
     ```json
     {
        'ip':'192.168.1.10',      # TCP IP address
        'port':'5000',            # TCP port number
        'udp_ip':'192.168.1.100', # UDP IP address
        'udp_port':'6000',        # UDP port number
        'udp_cycle':'20', # UDP cycle (period = udp_cycle * 5 ms)
     }
     ```
     
     </td>
     </tr>
     </tbody>
     </table>

   - Return value:
   
     An `ArmInterface` instance if creation succeeds, otherwise `None`.

2. Create a robotic arm instance using an ArmFactoryParam object.

   - Method:
   
     ```python
     create_arm(param: ArmFactoryParam) -> ArmInterface
     ```
   
   - Parameters:
   
     | Parameter | Description |
     |:---|:---|
     | `param` | Robotic arm configuration parameters. For details, please refer to [ArmFactoryParam](#ArmFactoryParam). |
   
   - Return value:
   
     An `ArmInterface` instance if creation succeeds, otherwise `None`.

3. Create a robotic arm instance from a configuration file.

   - Method:
   
     ```python
     create_arm_from_config(config_path: str) -> ArmInterface
     ```
   
   - Parameter:

      <table>
      <thead>
      <tr>
      <th>Parameter</th>
      <th>Description</th>
      </tr>
      </thead>
      <tbody>
      <tr>
      <td>
      
      `config_path`
      
      </td>
      <td>
      
      Path to the configuration file. The configuration file must be a JSON file.
      
      ```json
      {
          "arm_type":"xxx", # Robotic arm vendor
          "arm_name":"left_arm", # Robotic arm name
          "ip": "192.168.1.18", # TCP IP address
          "port": "8080", # TCP port number
          "udp_ip": "192.168.1.102", # UDP IP address
          "udp_port": "8089", # UDP port number
          "udp_cycle": "1", # UDP cycle (period = udp_cycle * 5 ms)
          "enable_gripper": "false" # Enable gripper
      }
      ```     
      </td>
      </tr>
      </tbody>
      </table>

   - Return value:
   
     An `ArmInterface` instance if creation succeeds, otherwise `None`.

### Angle Pass-through

Joint angles are sent to the robotic arm via CAN FD (Controller Area Network Flexible Data-rate), bypassing trajectory planning by the controller. If the method is called correctly, the robotic arm executes the motion immediately.

<Aside>
- The pass-through control performance is affected by the communication cycle and trajectory smoothness; therefore, a stable communication cycle is required to avoid large fluctuations.
- When using this feature, it is recommended to plan smooth trajectories to ensure stable operation of the robotic arm.
- The I-series wired Ethernet interface supports a minimum cycle time of 2 ms, providing higher real-time performance.
</Aside>

- Method:

  ```python
  move_joint_position_canfd(
      self,
      joints: list[float],
      follow: bool,
      trajectory_mode: int = 0,
      radio: int = 0
  ) -> alphabot.core.AbcErrorCode
  ```

- Parameters:

  | Parameter | Description |
  |:---|:---|
  | `joints` | Target joint positions, in degrees. |
  | `follow` | Follow mode flag. Accepts `True` or `False`.<br/>- `True`: High-follow mode.<br/>- `False`: Low-follow mode.<br/>Note: High-follow mode requires a pass-through cycle of 10ms or less. |
  | `trajectory_mode` | Used in high-follow mode. Accepts 0, 1, or 2.<br/>- `0`: Full pass-through mode.<br/>- `1`: Curve-fitting mode.<br/>- `2`: Filtering mode. |
  | `radio` | - In curve-fitting mode: `radio` is the smoothing coefficient, range 0-100.<br/>- In filtering mode: `radio` is the filter parameter, range 0-1000. |

- Return value:

  `AbcErrorCode.SUCCESS` indicates success. For other error codes, please refer to [AbcErrorCode](/en/sdk/python/data-structures#AbcErrorCode).

### Pose Pass-through

The target pose is sent to the robotic arm via CAN FD (Controller Area Network Flexible Data-rate), bypassing trajectory planning by the controller.

When the target pose is received by the arm controller, the controller first attempts inverse kinematics calculation. If the inverse kinematics solution succeeds and the calculated joint angles do not differ significantly from the current joint positions, the angles are sent directly to the joints for execution, skipping additional trajectory planning.

This feature is suitable for scenarios that require periodic pose adjustments, such as visual servoing applications.

<Aside>
- The pass-through control performance is affected by the communication cycle and trajectory smoothness; therefore, a stable communication cycle is required to avoid large fluctuations.
- When using this feature, it is recommended to plan smooth trajectories to ensure stable operation of the robotic arm.
- The I-series wired Ethernet interface supports a minimum cycle time of 2 ms, providing higher real-time performance.
</Aside>

- Method:

  ```python
  move_cartesian_pose_canfd(
      self,
      pose: alphabot.core.Pose,
      follow: bool,
      trajectory_mode: int = 0,
      radio: int = 0
  ) -> alphabot.core.AbcErrorCode
  ```

- Parameters:

  | Parameter | Description |
  |:---|:---|
  | `pose` | Target pose. |
  | `follow` | Follow mode flag. Accepts `True` or `False`.<br/>- `True`: High-follow mode.<br/>- `False`: Low-follow mode.<br/>Note: High-follow mode requires a pass-through cycle of 10ms or less. |
  | `trajectory_mode` | Used in high-follow mode. Accepts 0, 1, or 2.<br/>- `0`: Full pass-through mode.<br/>- `1`: Curve-fitting mode.<br/>- `2`: Filtering mode. |
  | `radio` | - In curve-fitting mode: `radio` is the smoothing coefficient, range 0-100.<br/>- In filtering mode: `radio` is the filter parameter, range 0-1000. |

- Return value:

  `AbcErrorCode.SUCCESS` indicates success. For other error codes, please refer to [AbcErrorCode](/en/sdk/python/data-structures#AbcErrorCode).

### Set Gripper Aperture Position

After the gripper reaches the target position, the following behavior applies:

- If the current aperture is smaller than the target, the gripper opens to the target position at the default speed.
- If the current aperture is larger than the target, the gripper closes towards the target position at the specified speed and torque limit. The gripper stops when either the torque threshold is exceeded or the target position is reached.

- Method:

  ```python
  set_gripper_position(
      self,
      position: int,
      block: bool,
      timeout: int
  ) -> alphabot.core.AbcErrorCode
  ```

- Parameters:

  | Parameter | Description |
  |:---|:---|
  | `position` | Target gripper aperture position. Valid range: 1-1000 (dimensionless). |
  | `block` | Blocking mode flag.<br/>- `true`: Blocking mode. Waits for the controller to confirm the gripper has reached the position.<br/>- `false`: Non-blocking mode. Does not wait for the gripper-in-position feedback. |
  | `timeout` | Timeout behavior depends on the blocking mode.<br/>- Blocking mode: Maximum waiting time for the gripper to reach the target position, in seconds.<br/>- Non-blocking mode: If set to 0, the method returns immediately after sending the command; otherwise, it returns after receiving the successful setting command. |

- Return value:

  `AbcErrorCode.SUCCESS` indicates success. For other error codes, please refer to [AbcErrorCode](/en/sdk/python/data-structures#AbcErrorCode).

### Configure Modbus RTU Mode

After the robotic arm is started, you must first call the following method to configure Modbus RTU mode before performing any operation on the communication port. Otherwise, an error will be returned.

The robotic arm persists the Modbus configuration. After a reboot, the arm automatically restores the mode that was configured before power-off.

- Method:

  ```python
  set_modbus_mode(
      self,
      port: int,
      baudrate: int,
      timeout: int
  ) -> alphabot.core.AbcErrorCode
  ```

- Parameters:

  | Parameter | Description |
  |:---|:---|
  | `port` | Specifies the communication port. Valid values: 0, 1, or 2.<br/>- `0`: Controller RS485 port as RTU master.<br/>- `1`: End interface board RS485 port as RTU master.<br/>- `2`: Controller RS485 port as RTU slave. |
  | `baudrate` | Baud rate, supporting three common baud rates, namely 9600, 115200, and 460800. |
  | `timeout` | Timeout period, in 100 ms.<br/>For all read and write commands to the Modbus device, if no response data is returned within the specified timeout period, a timeout error will be returned.<br/>The timeout period cannot be set to 0; if set to 0, the robotic arm will take it as 1 for configuration. |

  <Aside>
  Other configurations are as follows by default: data bit - 8, stop bit - 1, parity check - no.
  </Aside>

- Return value:

  `AbcErrorCode.SUCCESS` indicates success. For other error codes, please refer to [AbcErrorCode](/en/sdk/python/data-structures#AbcErrorCode).
